# 长视频准实时分析落地方案

## 1. 目标定义

本项目的目标不是“直播级 1-3 秒内回答任意复杂问题”，而是实现工程上可落地的“准实时”：

- 对 1 小时左右视频，在索引阶段达到 `20-50 分钟内可问`
- 在索引完成后，用户发起一次 QA 时：
  - 音频型问题：`3-12 秒`返回答案
  - 视觉型问题：`20-90 秒`返回答案
  - 检索本身：`毫秒到 1 秒`

这里的核心思想是：

`轻量预处理/建库 -> 轻量路由 -> 双路粗检索 -> 小窗口精看 -> 大模型回答`

而不是：

`压缩后把整段视频继续全量送入大模型`

## 2. 当前系统现状

当前仓库的主流程仍然偏离线批处理：

- 全量执行 `ASR + 切段 + 抽帧 + OCR`
- 对每个 chunk 跑一次 Qwen-VL 生成 `chunk card`
- 之后再做 section/global/thinking
- QA 主要基于文本证据，不是命中后再对局部视频做精看

这条链路的主要问题是：

- 大模型参与得太早，且参与范围太大
- 视觉侧没有独立的轻量索引链路
- QA 没有显式的 `audio / video / mixed` 路由
- 服务层仍然按单 worker 保守运行，不适合低时延问答

## 3. 总体改造原则

### 3.1 保留的部分

以下模块可以继续复用：

- `long_video_pipeline/asr_ocr.py`
- `long_video_pipeline/video_ops.py`
- `long_video_pipeline/service.py`
- `long_video_pipeline/settings.py`
- `long_video_pipeline/model_runtime.py`

### 3.2 需要弱化的部分

以下能力不再作为“准实时主链路”的必经阶段：

- 全量 `chunk card` 生成
- 全量 `section summary`
- 全量 `global summary`
- 全量 `thinking review`

这些功能可以保留为：

- 离线增强模式
- 任务完成后的补充摘要模式
- 高质量报告导出模式

### 3.3 准实时链路的关键原则

- 先建轻量索引，再做精分析
- 让大模型只看命中的小窗口，不看整片
- 音频问题优先走文本/音频链路
- 视觉问题优先走视觉链路
- 混合问题双路都查，再融合排序

## 4. 目标架构

### 4.1 输入阶段

用户上传视频后，系统先做以下轻量处理：

1. 视频与音频解复用
2. 音频轨压缩为适合 ASR 的格式
3. 视频轨压缩为适合视觉 embedding 的格式
4. 并行建立音频索引和视觉索引

### 4.2 音频索引链路

音频侧目标是快速回答：

- “他们说了什么”
- “某个人提到了什么”
- “这一段讨论了什么主题”

建议流程：

1. 使用 `ffmpeg` 将音轨转为 `16kHz mono`
2. 使用 `VAD` 做语音活动检测，切出连续语音段
3. 用 `Faster-Whisper` 做带时间戳 ASR
4. 将 ASR 文本按时间片组织为检索单元
5. 使用轻量 embedding 模型建音频文本向量库

建议模型：

- ASR：`faster-whisper small / medium / large-v3` 按速度和精度分级
- 文本 embedding：`bge-small-zh`、`bge-m3` 或现有轻量模型

注意：

- “声音异常”不能只靠 ASR 文本
- 若后续要支持尖叫、警报、撞击、玻璃破碎等，需要单独引入音频事件检测模型

### 4.3 视觉索引链路

视觉侧目标是快速回答：

- “谁拿走了文件”
- “什么时候有人进入房间”
- “什么时候画面出现白板/手机/文件”

建议流程：

1. 对视频轨做低分辨率解码
2. 先做镜头边界检测或运动峰值检测
3. 在镜头变化点和固定频率点上采样
4. 每秒采样 1 帧作为基础方案
5. 对采样帧提取视觉 embedding
6. 写入视觉向量库

建议模型：

- 视觉 embedding：`SigLIP`
- 向量库：单机优先 `FAISS`，后续再考虑 `Milvus`

注意：

- 仅靠固定 `1fps` 容易漏掉短时动作
- 应补充：
  - 镜头切换采样
  - 运动峰值补帧
  - 目标出现突变时的额外采样

### 4.4 查询路由链路

用户提问后，不应直接进入统一 QA。

应先执行轻量路由：

1. 判断问题更偏 `audio`、`video` 还是 `mixed`
2. 判断问题意图：
   - 对话内容
   - 人物动作
   - 物体状态变化
   - 时间定位
   - 综合问答
3. 输出检索策略和权重

示例：

- “那两个人在办公室谈了什么” -> `audio: 0.8, video: 0.2`
- “谁把桌上的文件拿走了” -> `video: 0.85, audio: 0.15`
- “他拿文件前后说了什么” -> `audio: 0.5, video: 0.5`

第一版不必强依赖小 LLM，可以先做：

- 规则模板
- 关键词词典
- 小分类器

只有在规则无法稳定覆盖时，再引入轻量 LLM 做路由补强。

### 4.5 粗检索链路

根据路由结果执行双路粗检索：

- 音频库检索：返回相关文本段及时间范围
- 视觉库检索：返回相关帧及时间范围
- 若为 mixed 问题：双路都查，并融合排序

粗检索目标不是直接回答，而是把 1 小时视频缩小到若干个小窗口，例如：

- Top 3 时间窗
- 每个窗口 `30-120 秒`

### 4.6 精分析链路

对粗检索命中的时间窗，执行二次精看：

1. 回源到原视频
2. 对命中窗口进行密集抽帧
3. 按需附加该窗口的 ASR/OCR 文本
4. 将“高价值帧 + 文本上下文”送入大 VLM
5. 由大 VLM 输出最终答案

这里的大模型输入不应是“整段长视频”，而应是：

- 8-32 张高价值帧
- 对应的 ASR 文本
- 必要时附带 OCR
- 明确的问题和输出模板

这一步才是高成本步骤，因此必须发生在粗检索之后。

## 5. 推荐的系统分层

### 5.1 在线主链路

在线主链路只保留以下阶段：

1. `ingest`
2. `audio indexing`
3. `visual indexing`
4. `query routing`
5. `coarse retrieval`
6. `window refinement`
7. `vlm answer`

### 5.2 离线增强链路

原有的全片摘要能力改为离线增强链路：

1. `chunk card`
2. `section summary`
3. `global summary`
4. `thinking review`

触发方式：

- 用户主动选择“生成完整报告”
- 夜间后台批处理
- 完成索引后低优先级补算

## 6. 性能预算

以下是 1 小时视频的工程目标预算。

### 6.1 索引阶段预算

- 解复用：`1-3 分钟`
- 音频压缩 + VAD：`2-5 分钟`
- ASR：`10-25 分钟`
- 视频低清解码 + 采样：`5-10 分钟`
- 视觉 embedding + 建库：`5-12 分钟`
- 向量库落盘：`1-3 分钟`

整体目标：

- 理想目标：`15-20 分钟`
- 现实目标：`20-30 分钟`
- 保守上限：`35-50 分钟`

### 6.2 QA 阶段预算

- 路由：`50-300ms`
- 粗检索：`50-800ms`
- 时间窗合并与重排：`100-500ms`
- 小窗口密集抽帧：`2-10 秒`
- 大 VLM 精看与回答：`5-60 秒`

最终用户感知：

- 纯音频问题：`3-12 秒`
- 一般视觉问题：`20-45 秒`
- 复杂跨模态问题：`30-90 秒`

## 7. 与当前代码结构的对应关系

### 7.1 现有模块改造方向

#### `long_video_pipeline/asr_ocr.py`

保留并扩展为：

- 音频解复用
- 音频标准化
- VAD 分段
- Faster-Whisper 转写
- 仍保留 OCR，但不再把 OCR 当主链路核心

#### `long_video_pipeline/video_ops.py`

扩展为：

- 视频解复用与压缩
- 低清采样
- 镜头检测
- 运动峰值采样
- 小窗口密集抽帧

#### `long_video_pipeline/retrieval.py`

拆分为两层更合理：

- `audio_retrieval`
- `visual_retrieval`
- `fusion_retrieval`

如果不想立刻拆文件，至少要在当前文件中把逻辑拆成三类接口。

#### `long_video_pipeline/pipeline.py`

需要重构为两条模式：

- `realtime_index_pipeline`
- `offline_summary_pipeline`

不要让准实时主链路依赖全量摘要链路。

#### `long_video_pipeline/service.py`

新增能力：

- 查询路由接口
- 双索引状态查询
- 命中窗口精看接口
- 缓存最近问题的答案和中间结果

### 7.2 建议新增模块

建议新增：

- `long_video_pipeline/routing.py`
- `long_video_pipeline/audio_index.py`
- `long_video_pipeline/visual_index.py`
- `long_video_pipeline/refine.py`

职责如下：

- `routing.py`：问题分类与模态权重判断
- `audio_index.py`：音频文本索引的构建与查询
- `visual_index.py`：视觉向量构建与查询
- `refine.py`：命中窗口回源、密集抽帧、VLM 精看

## 8. 分阶段实施计划

### Phase 0：准实时骨架改造

目标：

- 从现有“全量摘要先行”中抽出可独立运行的“索引先行”链路

任务：

- 拆分 `pipeline.py`
- 明确在线索引输出格式
- 定义音频索引和视觉索引的统一时间片结构

交付结果：

- 视频上传后能只跑索引，不跑全量 VLM 摘要

### Phase 1：音频优先准实时上线

目标：

- 先让“谈了什么”类问题快起来

任务：

- 音频解复用
- VAD + Faster-Whisper
- 文本 embedding 建库
- QA 先走音频文本召回

交付结果：

- 支持音频型问题在 `3-12 秒`内回答

### Phase 2：视觉粗检索上线

目标：

- 让“谁拿了什么”“什么时候发生了什么”类问题具备粗定位能力

任务：

- 低清采样
- SigLIP embedding
- FAISS 建库
- 视觉问题先检索再回源

交付结果：

- 支持视觉问题定位到 `30-120 秒`窗口

### Phase 3：命中窗口精看

目标：

- 在粗定位基础上给出更可信的最终答案

任务：

- 小窗口密集抽帧
- 帧选择去重
- VLM 精看 prompt 设计
- 结果缓存

交付结果：

- 视觉/混合问题回答质量明显提升

### Phase 4：混合问题与性能打磨

目标：

- 提升复杂跨模态问题的稳定性

任务：

- 双路融合排序
- 路由器优化
- 常见问题缓存
- 低延迟批量推理
- 服务并发与资源控制

交付结果：

- 系统进入稳定准实时阶段

## 9. 第一版实现建议

第一版不要追求过度复杂，建议按最小可行集启动：

### 必做

- 音视频解复用
- 音频 ASR 索引
- 视频 `1fps` + SigLIP 索引
- 问题路由
- Top-K 时间窗召回
- 小窗口密集抽帧
- 大模型精看回答

### 暂缓

- 全量全片摘要必跑
- Milvus
- 多模型自动切换
- 复杂音频事件识别
- 多摄像头联合分析

## 10. 风险与注意事项

### 10.1 主要风险

- `1fps` 对短动作召回不稳定
- 仅靠 ASR 无法覆盖声音异常
- 路由器误判会导致召回路径偏离
- 大 VLM 精看仍然是系统最高时延来源
- 若继续把完整报告生成放在主链路中，准实时目标会直接失效

### 10.2 对应缓解策略

- 加镜头变化点采样和运动峰值补帧
- 为声音异常预留独立音频事件模型接口
- 路由输出权重而不是单一路径
- 对命中窗口长度设硬上限
- 将全片摘要下沉到离线增强模式

## 11. 最终判断

这套方案可以把当前项目从“离线视频总结系统”改造成“准实时视频问答系统”，前提是做对两件事：

1. 大模型退出全量主链路，只负责命中窗口精看
2. 音频和视觉分别建立轻量索引，并在查询阶段做路由与融合

若仍保留“全片先跑大模型，再提供 QA”的主流程，则无法达到准实时目标。
